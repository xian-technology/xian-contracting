# Shielded Hash Redesign — Poseidon-BN254 (F-1 remediation)

Status: **Implemented** — Poseidon-BN254 landed with the v4/v5 circuit ids
(`shielded-*-v4`, `shielded-command-*-v5`); kept as the design record for the
remediation.
Author: Security audit (2026-06-06)
Scope: replaces the unsound MiMC construction used by the shielded-note and
shielded-command circuits, contracts, and tooling.
Breaking: **yes** — consensus, circuit, and state-format breaking. The network
is pre-launch, so this is a clean re-version with **no backward compatibility
and no state migration**.

---

## 1. Background — why this redesign exists

The shielded stack hashes commitments, nullifiers, Merkle nodes, and command
bindings with a custom MiMC construction:

```
perm(state)        = iterate 91 rounds of  state := (state + c_round)^7
hash_many(values)  = state := 0; for v in values: state := perm(state + v); return state
```

This is **not a secure hash**. It has no Miyaguchi–Preneel/Davies–Meyer
feed-forward and a width-1 state (capacity 0), so:

- `perm` is an **efficiently invertible bijection** (`gcd(7, p-1) = 1`), hence
  `hash_many` is **not one-way**.
- The compression has **no capacity**, so it is **not collision-resistant**.

Confirmed impact (working PoC): a single note commitment can be re-opened to a
**different amount** with a valid recovered owner secret, and Merkle parents
collide with forward-only effort. Net effect: **unlimited value inflation /
draining of the entire shielded pool.** All in-circuit value-conservation and
64-bit range checks are correct but rest on a non-binding commitment, so they do
not help. Contract-level nullifier/`note_exists`/root-window defenses are correct
but bypassed (the attack uses a fresh valid nullifier and a genuinely-present
commitment).

The flaw is reimplemented identically in:

1. native Rust — `packages/xian-zk/src/shielded_notes.rs` (`mimc_*_native`)
2. the R1CS gadget — same file (`mimc_*_var`)
3. `con_shielded_note_token.py` and `con_shielded_commands.py` (Python `mimc_*`)

This redesign removes the root cause **and** the structural reason it could
exist in three places at once.

---

## 2. Goals and non-goals

**Goals**

- Replace the hash with a vetted, ZK-native construction whose commitment is
  **binding** and whose compression is **collision-resistant and one-way**.
- Establish a **single source of truth** for the hash (native Rust), with the
  R1CS gadget proven equal to it, and every other consumer delegating to it.
- Add explicit **domain separation** so cross-context and cross-arity collisions
  are impossible by construction.
- Re-version all circuits and verifying keys; ship clean, parity-tested code.

**Non-goals**

- No backward compatibility, no migration of existing notes (pre-launch).
- The MPC trusted-setup ceremony itself (operational; see F-2 in the task
  breakdown). This doc makes the setup *re-runnable and clearly gated*; running
  a real ceremony is a separate deliverable before mainnet.

## 3. Chosen primitive — Poseidon over BN254 (arkworks)

**Decision:** use **Poseidon** on the BN254 scalar field via the audited
`ark-crypto-primitives` sponge, for both the native hash and the R1CS gadget.

Rationale:

- Poseidon is purpose-built for SNARK circuits, widely deployed and audited, and
  far cheaper in-circuit than 91-round MiMC-x⁷.
- `ark-crypto-primitives` provides `PoseidonSponge` (native) **and**
  `PoseidonSpongeVar` (R1CS) from the **same parameters**, so native and gadget
  are guaranteed to agree — eliminating the hand-written gadget that diverged in
  the MiMC design.
- We confirmed `gcd(5, p-1) = 1`, so the Poseidon S-box `x^5` is a valid
  permutation on BN254 Fr.

**Why not circomlib-compatible Poseidon:** that choice is only advantageous when
Python/JS reimplement the permutation and need shared cross-language test
vectors. In this codebase the Python SDK already delegates to the native Rust
bindings, the JS wallet has no hash implementation, and the contracts will stop
reimplementing the hash (Section 5). With no foreign permutation to keep in
sync, arkworks-native Poseidon is simpler and lower-risk.

### 3.1 Parameters (to be pinned during implementation)

- Field: BN254 scalar field `Fr`.
- S-box: `α = 5`.
- Width `t = 3` (rate `r = 2`, capacity `c = 1`), 128-bit security target.
- Rounds: full `R_F = 8`, partial `R_P = 57` (standard for `t = 3, α = 5`).
- Round constants + MDS: generated by the standard Grain-LFSR procedure
  (arkworks `find_poseidon_ark_and_mds`). The exact generated constants are
  **frozen into a committed constants module** and covered by known-answer tests
  so they can never drift.

The implementation MUST pin one concrete `PoseidonConfig` and reuse it
everywhere. The config bytes (or their hash) are recorded in this doc once
generated.

### 3.2 Hash interface and domain separation

Define one variable-arity field hash on top of the sponge:

```
poseidon_hash(domain_tag, values) =
    sponge := PoseidonSponge::new(CONFIG)
    sponge.absorb(domain_tag)        # 1 field element, fixed per use
    sponge.absorb(values...)         # library handles rate/padding
    return sponge.squeeze_field_elements(1)[0]
```

Each logical use gets its own constant `domain_tag` (a distinct small field
element, e.g. a fixed enum index, documented in the constants module):

| Use | tag | inputs |
|---|---|---|
| `owner_public` | `DOM_OWNER` | `[owner_secret]` |
| `note_commitment` | `DOM_COMMIT` | `[asset_id, owner_public, amount, rho, blind]` |
| `nullifier` | `DOM_NULLIFIER` | `[asset_id, owner_secret, rho]` |
| `merkle_parent` | `DOM_MERKLE` | `[left, right]` |
| `command_nullifier_digest` | `DOM_CMD_NULL` | `input_nullifiers` |
| `command_binding` | `DOM_CMD_BIND` | binding fields |
| `command_execution_tag` | `DOM_CMD_TAG` | `[nullifier_digest, command_binding]` |

Domain tags bind both the context and (together with the sponge) the arity, so a
digest from one context can never be reinterpreted in another. This closes the
loose arity binding in the old design (where the input count was a separate
public input rather than part of the hash).

> Note: domain tags are an additive improvement over a plain re-hash. They are
> recommended and cheap; the binding/collision fix comes from Poseidon itself.

### 3.3 Required security property (becomes a test)

For a fixed `asset_id`, there must not exist two distinct openings
`(owner_secret, amount, rho, blind) != (owner_secret', amount', rho', blind')`
producing the same `note_commitment`. The F-1 PoC is inverted into a permanent
regression test asserting this is infeasible (Section 7).

## 4. Single-source-of-truth architecture

```
                         ┌──────────────────────────────────────┐
                         │  packages/xian-zk/src/                │
                         │   poseidon.rs   (CONFIG + tags)       │  <-- the ONLY
                         │   shielded_notes.rs                   │      hash
                         │     native helpers  (PoseidonSponge)  │      definition
                         │     gadget helpers  (PoseidonSpongeVar)│
                         └───────────────┬──────────────────────┘
                                         │ pyo3
        ┌────────────────────────────────┼─────────────────────────────┐
        │                                 │                             │
  zk stdlib bridge                  xian_zk SDK (python)          (wallet/js)
  (contracting/stdlib/bridge/zk.py) already delegates → no hash    no hash impl
        │                              reimplementation             (delegates)
        │
  shielded contracts (con_shielded_note_token / con_shielded_commands)
   - NO python mimc/poseidon permutation
   - consensus derivations via zk.shielded_* bridge calls
   - ZERO_HASHES = regenerated constants (parity-tested), not computed on-chain
```

Principle: **the native crate is the only place a permutation is implemented.**
The gadget is proven equal to it by test. Everyone else calls in or embeds
parity-tested constant data only.

### 5.3 `zk` stdlib bridge (`xian-contracting`)

  commitment/nullifier/merkle helpers already are via `shielded_*`). Add only

  change to metering except recosting if a new helper is added.

## 6. Trusted setup (interaction with F-2)

New circuits ⇒ new proving/verifying keys ⇒ setup must be re-run regardless.
This redesign:

- keeps an explicit `insecure-dev` (fixed-seed) path for tests only, clearly
  labelled;
- keeps a `single-party` (OS-rng) path for testnet, clearly labelled
  non-production;
- structures bundle/manifest generation so an **MPC ceremony** output can be
  dropped in for mainnet.

Running the ceremony is tracked separately (F-2). No mainnet deployment of these
keys until that is done.

---

## 7. Test strategy (the safety gate)

1. **Known-answer tests** — pin Poseidon KAT vectors for the chosen config;
   assert native output matches them. Detects any accidental param change.
2. **Native ⇄ gadget parity** — for random inputs at every arity used, assert
   the gadget witness equals the native value inside a satisfied constraint
   system. (This is the check the MiMC design lacked teeth on.)
3. **Binding regression (F-1 inverted)** — attempt to construct a second opening
   of a fixed commitment with a different amount; assert infeasible. Attempt the
   forward-only `merkle_parent` collision; assert it fails. These stay forever.
4. **Cross-layer parity** — contract `ZERO_HASHES`/`ZERO_ROOT` equal
   `zk.shielded_note_*`; SDK commitment/nullifier equal native; regenerate and
   diff `tests/fixtures/*.json`.
5. **End-to-end** — deposit → transfer → withdraw and the command flow verify
   against the freshly re-run setup; a deliberately inflated witness **fails**
   verification.
6. **Negative/encoding** — existing canonical-field-element and length checks
   still pass; non-canonical inputs still rejected.

A change is not "done" until 1–6 are green.

## 8. Rollout order

1. Land `poseidon.rs` + KATs (Section 7.1) on a branch.
2. Re-point native helpers; add native⇄gadget parity (7.2) and binding
   regression (7.3).
3. Regenerate zero-hash constants + bundles + fixtures; update contracts to drop
   Python hash and dead code; cross-layer parity (7.4).
4. Re-run setup (interim single-party); register v4/v5 keys; e2e (7.5).
5. Update docs (`stdlib/zk.md`, `concepts/shielded-zk-stack.md`) to describe the
   new construction and the setup/registry trust consequences.
6. External cryptographic review of `poseidon.rs` params + circuits before
   mainnet; MPC ceremony (F-2); then production key registration.

## 9. Risk notes

- **Primary risk** is parameter/encoding consistency. Mitigated by using one
  arkworks config for native+gadget, KATs, and the parity suite — and by having
  no foreign permutation reimplementations.
- **Performance**: Poseidon is cheaper in-circuit than the old MiMC; on-chain the
  contracts no longer hash at all (constants only), so chi cost should drop. Re-cost the
  `zk` helpers if any new one is added.
- **Scope discipline**: circuit *logic* (conservation, ranges, membership,
  enabled-slot handling) is intentionally untouched; only the hash underneath it
  changes. This keeps the diff reviewable.
